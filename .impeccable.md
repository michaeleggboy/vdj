## Design Context

### Users

- **Who**: People experimenting with **gesture-controlled mixing**—hobby DJs, makers, and developers—not a mass-market consumer DJ app.
- **Context**: **Local-first**: webcam + Python hand service on loopback + browser (`web/`). Often a **desk setup** with the camera seeing their hands over a clear surface; may be low light (club-ish room).
- **Job to be done**: **Load two tracks**, **blend them** with crossfader and per-deck levels **without touching the mouse** for the main mix, optionally **calibrate** neutral pose and ranges. Secondary: theme tweaks, audio enable, file pickers.

### Brand Personality

- **Three words**: **Tactile**, **local**, **hardware-literate** — the UI should read as a **credible tabletop / controller metaphor**, not a generic web dashboard or a toy game.
- **Voice**: Short labels (e.g. “Level”, “Crossfader”), technical calibration copy where needed; avoid cutesy microcopy.
- **Emotional goal**: **Confidence and control**—clear deck identity (A/B), visible hands overlay, predictable faders; a hint of **craft** (wood-leaning mixer strip, jog elevation) without skeuomorphic clutter.

### Aesthetic Direction

- **Visual tone**: **Dark, focused “tabletop”**—warm graphite / charcoal room, **wood-accent mixer column**, **large platters** with vinyl/jog metaphors. Deck accents are **distinct but muted** by default (**hardware** preset: teal-grey vs mauve; **classic** preset: brighter teal/magenta).
- **Tech stack (UI)**: React + Vite + **plain CSS** (`web/src/App.css`); **no component library**. Theming uses **`--vdj-*` custom properties** on `:root`, overridden at runtime via `themeStore` + **Theme** panel (persisted in `localStorage`).
- **Typography**: `system-ui`, "Segoe UI", sans-serif—functional, not expressive display type. Hierarchy via size, weight, letter-spacing, opacity—not decorative fonts unless deliberately chosen later.
- **References (intent)**: Physical **2-deck + mixer** layout (e.g. DJ controller faceplate)—**hierarchy** and **depth** cues, not pixel-perfect clone of any brand.
- **Anti-references**: Avoid **generic SaaS** layouts (dense card grids, purple-blue gradients, “AI default” aesthetics). Avoid **playful gamification** that undermines serious mixing. Don’t **wash out** deck or hand colors—hands must stay readable over platters (`z-index` + contrast).

### Design Principles

1. **Hands-first legibility** — Overlays and controls must not hide critical feedback; platters spin and **pause with intent** when a hand is on a deck; faders remain **pointer-accessible** where needed.
2. **Hardware metaphor, browser reality** — Skeuomorphism **serves recognition** (jog shadow, wood strip); don’t add chrome that fights performance or accessibility.
3. **Token-driven color** — Prefer **`var(--vdj-…)`** (and `rgba(var(--vdj-deck-*-rgb), …)`) so **Theme** presets and user overrides stay coherent; new UI should extend tokens, not hardcode one-off hexes.
4. **Motion with restraint** — Platters animate spin; **respect `prefers-reduced-motion`** for spin speed and banner animation. Avoid gratuitous motion on every interaction.
5. **Local & honest** — Copy and UI should reinforce **on-device processing** (no cloud handoff theater); calibration affordances (**Neutral**, two-point snaps) stay **discoverable** without cluttering the main tabletop.

### Accessibility & inclusion (current baseline)

- **Reduced motion**: Partially addressed in CSS (e.g. platter duration, neutral banner digit); extend when adding new animations.
- **Focus**: Visible **focus outlines** on buttons tied to **accent** RGB token.
- **Formal WCAG target**: Not locked in repo—when polishing, **check contrast** on `--vdj-fg` vs surfaces and on deck/hand colors; don’t rely on color alone for state where possible.
